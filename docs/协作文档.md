## 一、项目简介

基于 Vue 3 + TypeScript 的水电站闸门智能调度 Web 平台，前端负责可视化展示与管理配置，**不参与实时 AI 推理和控制指令下发**。

| 类别 | 选型 |
|------|------|
| 框架 | Vue 3（`<script setup lang="ts">`） |
| 构建 | Vite |
| UI | Element Plus |
| 状态 | Pinia |
| 路由 | Vue Router 4 |
| 样式 | SCSS + CSS Variables（`design-tokens.scss`） |
| 请求 | Axios |

## 二、环境准备

### 2.1 前置要求

- Node.js ≥ 18
- npm ≥ 9
- 编辑器推荐 VS Code / Cursor，并安装 ESLint、Vue - Official 插件

### 2.2 本地启动

```bash
git clone <仓库地址>
cd Water-conservancy
npm install
npm run dev
```

默认访问 `http://localhost:5173`（端口占用时会自动切换）。

### 2.3 常用命令

| 命令 | 说明 |
|------|------|
| `npm run dev` | 启动开发服务器 |
| `npm run build` | 类型检查 + 生产构建 |
| `npm run lint` | ESLint 检查并自动修复 |
| `npm run format` | Prettier 格式化 |
| `npm run preview` | 预览构建产物 |

### 2.4 环境变量

在项目根目录创建 `.env.local`（**不要提交到 Git**）：

```env
VITE_API_BASE_URL=/api
```

## 五、Git 协作流程

### 5.1 分支命名

```
feature/模块名-功能描述    例：feature/login-password-policy
fix/模块名-问题描述        例：fix/router-auth-redirect
refactor/范围-描述         例：refactor/api-request-layer
```

### 5.2 提交信息

```
<type>(<scope>): <subject>

type: feat | fix | refactor | docs | style | chore
scope: login | dashboard | warning | layout | api 等
```

示例：

```
feat(login): 实现密码强度校验与首次改密弹窗
fix(router): 修复未登录时跳转逻辑
docs(components): 登记 EmergencyStop 组件
```

### 5.3 合并流程

```
1. 从 main 拉最新代码
2. 创建 feature 分支开发
3. 本地 npm run lint && npm run build 通过
4. 提交 PR，填写变更说明和测试步骤
5. 至少 1 人 Code Review 通过后合并
6. 合并后删除 feature 分支
```

### 5.4 禁止事项

- 不要直接 push 到 `main`
- 不要 `--force` 推送
- 不要提交 `.env.local`、密钥、个人配置
- **不要擅自修改 `src/main.ts`**（应用入口，改动需提前沟通）

## 六、开发规范速查

> 完整规范见 [前端开发规范与最佳实践（优化版）.md](./前端开发规范与最佳实践（优化版）.md)，Code Review 不达标直接打回。

| 类别 | 要求 |
|------|------|
| 目录命名 | `kebab-case` |
| 组件文件 | `PascalCase.vue` |
| Composable | `use` 前缀 + `camelCase.ts` |
| 单文件行数 | Vue ≤ 600 行，Store ≤ 200 行 |
| 样式 | 必须 `scoped`；颜色/间距用 CSS 变量，禁止大量内联样式 |
| 类型 | 禁止 `as any` / `@ts-ignore` |
| 接口字段 | 必须来自后端文档，禁止自造 |
| 搜索 | 300ms 防抖 |
| 弹窗 | 必须有完整 Loading 态 |
| 状态 | 禁止跨模块直接改 Store，按业务域拆分 |

### 6.1 Vue 文件骨架

```vue
<script setup lang="ts">
// 1. 外部依赖
// 2. 类型定义
// 3. Props & Emits
// 4. Store / Composable
// 5. 响应式数据
// 6. Computed / Watch
// 7. 生命周期
// 8. 方法
</script>

<template>
  <!-- 模板 -->
</template>

<style scoped lang="scss">
/* 样式 */
</style>
```

## 七、文档协作要求

### 7.1 必须更新的文档

| 场景 | 更新文件 |
|------|----------|
| 新增/修改通用组件 | `docs/components.md` |
| 新增/修改接口 | `docs/api.md` |
| 架构级决策（换库、改目录结构等） | `docs/adr/xxx.md` |
| 认领/变更模块负责人 | 本文档 §4.1 / §4.2 |

### 7.2 通用组件登记条件

- 被 **≥2 个页面** 复用的组件 → 登记到 `components.md`
- 仅单页面使用的组件 → 放在对应 `views/模块/` 下，**不登记**

### 7.3 ADR 模板

```markdown
# ADR-NNN: 标题

## 状态
已采纳 / 已废弃

## 背景
为什么需要做这个决策

## 决策
选择了什么方案

## 后果
带来的影响和 trade-off
```

## 九、协作沟通

### 9.1 开工前

1. 在本文档 §4 认领模块，填写负责人
2. 阅读 [九大板块设计汇总](./九大板块设计汇总.md) 对应章节
3. 确认后端接口文档是否就绪（未就绪先用 mock，但字段结构需对齐）

### 9.2 开发中

- 公共组件 / Composable 需复用时，**先沟通再开发**，避免重复造轮子
- 遇到架构级问题（改目录、换 UI 库等），先写 ADR 讨论
- 阻塞问题及时同步，不要 silent 超过 1 天

### 9.3 提 PR 前自检

```
□ npm run lint 无报错
□ npm run build 通过
□ 新增组件已登记 components.md
□ 新增接口已登记 api.md
□ 无 console.log 残留
□ 无 as any / @ts-ignore
□ 样式使用 scoped + CSS 变量
□ 改动范围仅限自己负责的模块（不重构无关代码）
```
