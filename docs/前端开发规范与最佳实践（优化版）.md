# 前端开发规范与最佳实践（优化版）

> **说明**：本文档基于原始 28 条前端建议进行专业化优化、分类整理和内容补充。每条优化后建议末尾标注对应原文编号，便于追溯。

---

## 一、代码规范与工程约束

### 1.1 文件与目录命名规范
**对应原文：第 1 条**

- **唯一性**：同一项目内禁止出现同名文件（含不同目录），避免构建歧义和引用混乱。
- **命名风格**：统一使用当文件夹名为 `kebab-case` 时，**内部的组件文件名**建议使用 `PascalCase`（如 `UserProfile.tsx`），项目启动时由前端负责人确定并在 README 中固化，后续全员遵守。
- **目录结构**：按业务模块分层，如 `views/`（页面）、`components/`（组件）、`composables/`（组合式函数）、`stores/`（状态管理）、`utils/`（工具函数）、`api/`（接口层），禁止跨层引用。
- **补充建议**：建议在项目根目录放置 `.claude/settings.json` 或 `CLAUDE.md`，将命名规范写入 AI 助手的上下文约束，避免 AI 生成文件时偏离约定。

二、核心组织规律：按“类型”分层 + 按“功能/领域”聚合

这是目前最主流的两种思路，**推荐大项目采用“混合模式”**：

#### 1. 顶层按“类型”分层（适合中大型项目）
将文件按**技术角色**区分，保持职责单一，便于全局查找工具函数。

```
src/
├── components/     # 纯 UI 组件（无业务逻辑）
├── pages/          # 页面级入口（Next.js 常用，或叫 views/）
├── hooks/          # 自定义 React/Vue Hooks
├── utils/          # 纯工具函数（日期、格式化、校验）
├── services/       # API 请求层（与后端交互）
├── stores/         # 状态管理（Pinia/Redux/Zustand）
├── types/          # 全局 TypeScript 类型定义
├── assets/         # 静态资源（图片、字体、全局样式）
└── constants/      # 全局常量
```

#### 2. 业务模块内按“功能/领域”聚合（适合大型复杂项目）
将相关文件**就近放置**，减少跨目录的跳转。这是现在更推崇的 **Feature-based（按特性）** 结构：

```
src/
├── features/               # 核心业务功能
│   ├── auth/               # 认证模块（高内聚）
│   │   ├── components/     # 仅该模块使用的组件
│   │   ├── hooks/          # 仅该模块使用的 Hooks
│   │   ├── services/       # 仅该模块的 API
│   │   └── types/          # 仅该模块的类型
│   ├── dashboard/          # 仪表盘模块
│   └── payment/            # 支付模块
├── shared/                 # 跨模块共享的资源
│   ├── ui/                 # 基础 UI 组件库（Button/Input）
│   ├── lib/                # 底层工具库
│   └── config/             # 项目配置
```

### 三、特殊文件夹的命名字典（约定大于配置）

| 用途              | 推荐命名                       | 备注                                                         |
| :---------------- | :----------------------------- | :----------------------------------------------------------- |
| **路由页面**      | `pages/` 或 `routes/`          | Next.js 强制使用 `app/`，Vue Router 常用 `views/`            |
| **UI 基础组件**   | `ui/` 或 `base/`               | 存放 Button、Input 等原子级组件，区别于业务组件              |
| **布局组件**      | `layouts/`                     | 存放 Header、Sidebar、Footer 骨架                            |
| **上下文/提供者** | `providers/`                   | 存放 ThemeProvider、AuthProvider 等                          |
| **常量**          | `constants/`                   | 也可叫 `config/`，但建议 `constants` 存放死数据，`config` 存放环境变量 |
| **测试辅助**      | `mocks/` 或 `__mocks__`        | 存放模拟数据                                                 |
| **临时/废弃**     | `deprecated/` 或 `_deprecated` | 加下划线前缀在文件管理器中会置顶，提醒开发者规避             |

1.  **禁止出现复数与单数混用**：要么全 `component`（不推荐），要么全 `components`（推荐）。
2.  **路径不要超过 4 层**：如果嵌套超过 4 层，说明抽象层次出了问题，考虑提炼为独立模块。
3.  **配合别名（Alias）**：无论怎么命名，务必在 `vite.config` 或 `tsconfig.json` 中配置 `@` 指向 `src`。引用时使用 `@/components/button`，避免无休止的 `../../../`。
4.  **组件文件命名**：当文件夹名为 `kebab-case` 时，**内部的组件文件名**建议使用 `PascalCase`（如 `UserProfile.tsx`），这样在 IDE 的选项卡中能一目了然区分组件与普通工具函数。

### 五、总结推荐方案

如果是**新建项目**，直接按这个结构起步：

```
src/
├── features/          # 按业务模块划分（高内聚）
├── shared/            # 全局共享
│   ├── ui/            # 基础 UI
│   ├── utils/         # 工具函数
│   ├── api/           # 请求封装
│   └── types/         # 全局类型
├── app/               # 应用入口（路由、全局样式、Provider）
└── assets/            # 静态资源
```

**记住最关键的一条：** 命名规律没有绝对对错，**只要写在项目根目录的 `CONTRIBUTING.md` 文档里，并用 ESLint 插件（如 `eslint-plugin-folders`）做约束，它就是好规律。** 

eslint-plugin-folders：检查只检查文件夹名的命名规范，不检查文件名和代码规范

### 1.2 ESLint 与代码格式化
**对应原文：第 9 条**

- **强制启用 ESLint**：项目初始化即配置 ESLint，推荐使用 `@antfu/eslint-config` 或 `eslint-config-airbnb` + Vue 规则扩展。禁止任何形式的 `// eslint-disable` 注释（确需例外需提交 CR 由前端负责人批准）。
- **Prettier 配合**：建议 `.prettierrc` 统一配置：
  - `printWidth`: 100
  - `singleQuote`: true
  - `semi`: false
  - `trailingComma`: 'all'
- **Git Hooks**：强制配置 `husky` + `lint-staged`，提交前自动执行 `eslint --fix` 和 `prettier --write`，阻断不规范的代码进入仓库。

功能基本同cI门禁，但区别在于这个发生在个人的本地电脑上，而cI门禁大部分时候用于服务器主分支上

- **CI 门禁**：在 CI 流水线中加入 `eslint --max-warnings 0` 检查，警告即阻断合并。

CI 门禁：设置在代码合并时的自动检查，可以减少主分支被错误代码污染。



1.代码规范检查（同eslint）2.单元测试（确保新增代码没有破坏旧功能）3.构建编译（Build）确认代码能正常打包，没有语法错误或依赖缺失4.安全扫描：检查依赖包有没有已知漏洞。

### 1.3 样式规范
**对应原文：第 21 条**

- **禁止大量内联样式**：内联样式仅允许用于动态计算的样式值（如动态宽高、动画参数），且需在代码注释中说明原因。
- **设计 Token 集中管理**：在全局样式文件或 CSS 变量中统一定义并导出：
  
  ```scss
  // design-tokens.scss
  :root {
    --color-primary: #{$primary};
    --color-secondary: #{$secondary};
    --color-success: #{$success};
    --color-warning: #{$warning};
    --color-error: #{$error};
    --font-size-xs: 12px;
    --font-size-sm: 13px;
    --font-size-base: 14px;
    --font-size-lg: 16px;
    --font-size-xl: 20px;
    --spacing-xs: 4px;
    --spacing-sm: 8px;
    --spacing-md: 12px;
    --spacing-lg: 16px;
    --spacing-xl: 24px;
    --border-radius-sm: 4px;
    --border-radius-base: 6px;
    --border-radius-lg: 8px;
    --shadow-sm: 0 1px 2px rgba(0,0,0,0.05);
    --shadow-base: 0 2px 8px rgba(0,0,0,0.08);
    --shadow-lg: 0 4px 16px rgba(0,0,0,0.12);
  }
  ```
- **组件样式覆盖规范**：表格样式、按钮样式、弹窗样式、表单样式均需依据设计 Token 统一，禁止在业务组件中直接覆写第三方组件库的内部类名（应通过 `deep()` 限定作用域并在 CR 中说明理由）。

### 1.4 Vue 文件结构规范
**对应原文：第 22 条**

推荐标准文件骨架：

```vue
<script setup lang="ts">
// ── 1. 外部依赖导入 ──
import { ref, computed, watch, onMounted } from 'vue'
import { useRouter } from 'vue-router'

// ── 2. 类型/接口定义 ──
interface FormData { /* ... */ }

// ── 3. Props & Emits ──
const props = defineProps<{ /* ... */ }>()
const emit = defineEmits<{ /* ... */ }>()

// ── 4. Store / Composable ──
const userStore = useUserStore()

// ── 5. 响应式数据 ──
const loading = ref(false)
const formData = ref<FormData>({})

// ── 6. Computed ──
const isDisabled = computed(() => /* ... */)

// ── 7. Watch ──
watch(() => props.id, (val) => { /* ... */ })

// ── 8. 生命周期 ──
onMounted(() => { /* ... */ })

// ── 9. 方法函数 ──
async function handleSubmit() { /* ... */ }
</script>

<template>
  <!-- 模板内容 -->
</template>

<style scoped lang="scss">
/* 组件样式 */
</style>
```

```
<script setup>
// ── 1. 外部依赖导入 ──
import { ref, computed, watch, onMounted } from 'vue'
import { useRouter } from 'vue-router'

// ── 2. 类型/接口定义 ──
// (JS 中无需定义 interface，直接写业务对象即可)
// 例如: 默认表单数据结构
const defaultFormData = { name: '', age: 0 }

// ── 3. Props & Emits ──
// Props 使用运行时声明（对象写法）
const props = defineProps({
  id: {
    type: String,
    required: true
  },
  title: {
    type: String,
    default: ''
  }
})

// Emits 使用数组或对象写法（数组更简洁）
const emit = defineEmits(['update', 'submit', 'close'])

// ── 4. Store / Composable ──
// const userStore = useUserStore()  // 假设 store 存在

// ── 5. 响应式数据 ──
const loading = ref(false)
const formData = ref({ ...defaultFormData })

// ── 6. Computed ──
const isDisabled = computed(() => {
  // 例如：表单未填完时禁用
  return !formData.value.name || loading.value
})

// ── 7. Watch ──
watch(() => props.id, (newVal, oldVal) => {
  console.log('ID 变化了', newVal)
  // 在这里重新加载数据
})

// ── 8. 生命周期 ──
onMounted(() => {
  console.log('组件挂载完成')
  // 初始化请求数据
})

// ── 9. 方法函数 ──
async function handleSubmit() {
  if (isDisabled.value) return
  loading.value = true
  try {
    // 模拟提交逻辑
    await new Promise(resolve => setTimeout(resolve, 1000))
    emit('submit', formData.value)
  } finally {
    loading.value = false
  }
}

// 额外的工具函数
function resetForm() {
  formData.value = { ...defaultFormData }
}
</script>
```

**补充规范**：

- 模板中禁止出现复杂表达式，一律抽为 `computed`。
- `<style>` 必须添加 `scoped`，确需全局样式时使用 `:global()` 语法并注释说明。

## 二、组件化与复用策略

### 2.1 主文件代码量控制
**对应原文：第 5 条**

- **硬性约束**：单文件行数 ≤ **600 行**（不含注释和空行可放宽至 650 行）。超过即需拆分。

- **拆分策略（按优先级）**：
  
  1. 弹窗类 → 抽取为独立组件，放置于当前目录 `./components/` 下。
  2. 查询表单 → 抽取为 `SearchForm.vue`（要遵循命名规范。
  3. 复杂表格 → 抽取为 `DataTable.vue`（要遵循命名规范。
  4. 重复的 template 片段 → 抽取为子组件或 slot。
  
  #### slot
  
  在 Vue 里，**`slot`（插槽）就是组件模板中预留的“内容占位符”**。
  
  用一句大白话概括：**父组件在调用子组件时，写在子组件标签中间的那段 HTML 内容，最终会填到子组件内部 `<slot>` 标签所在的位置。**
  
  结合你贴的这条规范（**“重复的 template 片段 → 抽取为子组件或 slot”**）：
  
  ---
  
  ### 1. 为什么“抽取为 slot”能减少代码量？
  
  针对的是这种情况：**A 页面和 B 页面都有一段几乎一样的 HTML 结构，但里面的文字、图标或按钮稍有不同。**
  
  - **错误做法（复制粘贴）**：在两个页面里把这段 50 行的 `<div>` 结构各写一遍。结果就是总行数暴涨，且将来改样式要改两个地方。
  - **正确做法（用 slot）**：把这段重复的 **“外壳”（布局结构）** 抽成一个子组件，把**“变化的内容”**留在父组件里通过 `<slot>` 填进去。
  
  ---
  
  ### 2. 基础用法演示（帮你秒懂）
  
  假设你有个弹窗，不同页面弹窗的“标题”和“底部按钮”不一样，但“遮罩、圆角卡片、关闭图标”都一样。
  
  **抽离出的子组件 `ModalWrapper.vue`（只写骨架）：**
  ```vue
  <template>
    <div class="modal-mask">
      <div class="modal-card">
        <!-- 1. 固定不变的关闭按钮 -->
        <button class="close-btn">×</button>
        
        <!-- 2. 内容占位符：这里等待父组件塞内容进来 -->
        <slot></slot>
        
        <!-- 3. 底部也留一个占位符 -->
        <div class="footer">
          <slot name="footer"></slot>
        </div>
      </div>
    </div>
  </template>
  ```
  
  **父组件使用（把变化的内容填进去）：**
  ```vue
  <template>
    <ModalWrapper>
      <!-- 这个 <h2> 和 <p> 会自动替换掉子组件里的 <slot></slot> -->
      <h2>确认删除？</h2>
      <p>删除后数据不可恢复</p>
      
      <!-- 这个按钮会替换掉子组件里 <slot name="footer"></slot> -->
      <template #footer>
        <button class="danger">确认删除</button>
      </template>
    </ModalWrapper>
  </template>
  ```
  
  **结果**：你只在子组件里写了 **1 次** 弹窗外壳（50 行），却在 10 个不同页面复用了它。每个页面只需写 **5 行** 变化的内容，大大降低了主文件的行数，完美契合“≤600 行”的硬性约束。
  
  ---
  
  ### 3. 关键区分：什么时候用“子组件”，什么时候用“slot”？
  
  | 场景                                                         | 推荐方案                                | 原因                                         |
  | :----------------------------------------------------------- | :-------------------------------------- | :------------------------------------------- |
  | **某段逻辑（数据请求、表单校验）完全一样**                   | 抽取为**子组件**                        | 数据和 UI 都固化了，直接封装死即可。         |
  | **某段 UI 布局（盒子、卡片、弹窗外壳）一样，但内部内容不一样** | 抽取为**子组件 + 预留 `slot`**          | 把“盒子”复用起来，把“馅儿”留给父组件自己填。 |
  | **某段 UI 只有 80% 一样，剩下 20% 位置要插不同的东西**       | 抽取为**子组件 + 多个具名插槽（name）** | 像上面的例子，头部、底部、侧边各留一个口子。 |
  
  ---
  
  ### 4. 补充两个高阶概念
  
  既然问到 `slot`，顺便提两个在 CR（代码审查）中常被表扬的点：
  
  - **默认插槽 vs 具名插槽**：
    - `<slot>`（不带 name）是默认插槽，一个组件只能有 1 个。
    - `<slot name="header">` 是具名插槽，可以有多个，用 `#header` 传入。
  - **作用域插槽（Scoped Slots）**：如果子组件里的数据（比如表格行数据）想传给父组件使用，可以在 `<slot :row="data">` 上绑定属性，父组件用 `#default="{ row }"` 接收。这是高级封装（如高度定制化表格）的核心利器。
  
  ---
  
  ### 5. 规范落地建议
  
  在执行“拆分策略第 4 条”时，你的决策树应该是：
  
  1. 这段重复的 HTML **数据来源完全一样**吗？→ 是，拆成**普通子组件**。
  2. 这段重复的 HTML **数据来源不一样**，但**骨架一样**？→ 是，拆成**带 `<slot>` 的布局组件**。
  
  这样做之后，你的主文件（父页面）里只剩下“调用组件”和“传入插槽内容”的代码，行数自然就降下来了。
  
- **量化建议**：一个 `.vue` 文件如果同时包含 ≥2 个弹窗 或 ≥1 个弹窗 + ≥1 个复杂表单，应在开发前即规划拆分方案。

### 2.2 复用组件的前置沟通
**对应原文：第 4 条**

- **定义**：任何预计被 **≥2 个页面 / ≥2 个模块** 使用的组件，在开发前必须在团队群 / 周会中同步，同步内容包括：
  - 组件名称与用途
  - Props 接口定义
  - 使用场景
- **组件登记**：所有通用组件在 `docs/components.md` 中登记，包含路径、用途、Props 说明、使用示例。
- **反冗余机制**：Code Review 时若发现未事先沟通的独立实现与已有组件功能重叠 ≥70%，应要求重构为统一组件。

### 2.3 全局组件注册标准
**对应原文：第 11 条**

- **注册条件（同时满足）**：
  1. 在 **≥5 个不同页面/模块** 中使用
  2. 与具体业务逻辑解耦（无硬编码业务字段）
  3. 仅通过 Props / Slots 与父组件交互（不直接访问 Store 或特定 API）
- **负面清单（禁止注册为全局组件）**：
  - 包含具体业务字段的表单（如"用户筛选表单"）
  - 依赖特定 Store 的场景组件
  - 仅在 1-2 个页面使用的组件
- **注册流程**：开发者提出 → 前端负责人审核 → 登记到组件文档 → 在 `main.ts` 中注册。

### 2.4 通用组件与接口文档化
**对应原文：第 10 条**

- **文档要求**：在项目 `docs/` 目录下维护：
  - `components.md` — 通用组件清单（路径、Props/Emits/Slots、使用示例、负责人）
  - `api.md` — 通用接口清单（URL、Method、请求/响应字段、调用示例）
- **更新纪律**：每次新增或修改通用组件/接口时，**同 负责人内必须更新对应文档**，否则 Code Review 不予通过。
- **工具辅助**：可引入 Storybook 或 Histoire 用于组件的可视化文档。
- Storybook：一个开发工具，可以脱离主应用，单独渲染和测试每一个组件。1.在不运行整个应用的情况下，专注于单个组件的开发2.可以自动化测试：交互测试：模拟用户点击、输入等行为，验证组件功能是否正常。可访问性测试：自动检查组件是否符合WCAG无障碍标准。
  视觉测试：通过像素级的对比，及时发现UI渲染的意外变化3.生成文档：根据你的“故事”自动生成清晰、可交互的组件文档，团队其他成员（如产品、设计）可以直接在浏览器中查看和试用组件
- Histoire：同storybook，只是这个是vite和vue3专用的轻型框架，而stroybook比较通用且插件比较多且配置复杂

## 三、接口对接规范

### 3.1 接口对接与自测
**对应原文：第 3 条**

- **对接完成后必须自测**，验证清单：
  - [ ] 接口请求/响应字段与前端绑定字段一一对应

- **工具建议**：复杂场景编写 Playwright / Cypress E2E 用例覆盖关键路径。

### 3.2 搜索功能验证
**对应原文：第 13 条**

### 3.3 Loading 状态规范
**对应原文：第 15 条**

- **弹窗 Loading 必须覆盖以下场景**：
  
  | 场景 | Loading 要求 |
  |------|-------------|
  | 弹窗打开 → 加载详情数据 | 整体 Loading（骨架屏/Spin 组件） |
  | 弹窗内下拉框数据获取 | 下拉框自身 Loading 态 |
  | 点击确定 → 调接口 | 确定按钮 Loading + 禁用态 |
  | 弹窗关闭后再打开 | 清空旧数据 + 重新 Loading |
- **全局请求 Loading**：建议封装 Axios 拦截器统一处理请求 Loading 态（见 5.1 第 8 项），避免每个请求手动 `loading.value = true/false`。

### 3.4 字段来源规范
**对应原文：第 24 条**

- **字段来源优先级（由高到低）**：
  1. 接口文档（Swagger / Apifox / YApi 中的字段定义）
  2. 后端实际返回结果（接口联调时以实际返回为准，发现与文档不一致需反馈后端修正文档）
  3. 数据库设计文档
  4. 产品原型标注
  5. 业务确认记录
  
- **红线**：禁止前端根据 UI 样式自行编造字段名、禁止将 Mock 字段名当作真实接口字段使用、禁止在未与后端确认的情况下修改字段名。

- **类型安全**：建议使用 TypeScript /JavaScript定义接口返回类型，并通过工具（如 `openapi-typescript`）从 Swagger 文档自动生成类型定义，避免手写导致字段偏差。

- openapi-typescript 是一个命令行工具，它能将 OpenAPI 规范文件（JSON或YAML格式）自动转换成 TypeScript 类型定义。
  后端通常会提供一份 OpenAPI（原Swagger）规范文档来描述所有API的请求参数、响应结构等信息。以往，前端开发者需要根据这份文档手动编写TypeScript类型，这不仅繁琐、容易出错，而且一旦API发生变更，手动更新的成本很高。openapi-typescript 通过自动化方式解决了这个痛点，它从静态的 OpenAPI 文档中生成类型，整个过程仅需 Node.js 环境。
  生成类型：在项目根目录运行 npx 命令，指定 schema 文件路径和输出文件路径。生成的 schema.d.ts 文件就是一个包含所有API类型定义的TypeScript模块。

- Swagger 是一套用于设计、构建、文档化和使用 RESTful API 的“工具集”。有三个工具：Swagger UI：最核心、最直观的工具。它能把枯燥的 YAML/JSON 文档，渲染成一个在浏览器里可交互的 API 文档页面。在这个页面上，你可以看到所有接口的路径、参数说明，甚至可以直接点击“Try it out”按钮，在页面上模拟发送请求并查看返回结果，相当于一个“API 调试器”
  Swagger Editor：一个在线的 YAML/JSON 编辑器。它最大的特点是能实时预览 Swagger UI 效果，并在你编写时进行语法校验。后端开发人员通常用它来“起草”这份 API 文档。

  Swagger Codegen：一个代码生成器。它能根据 OpenAPI 规范，自动生成客户端 SDK（如 Java、Python、TypeScript 的 API 调用代码）或服务端骨架（如 Spring Boot 的 Controller 接口定义）。这和 openapi-typescript 功能类似，但 openapi-typescript 更轻量，只生成类型，不生成运行时逻辑。

## 四、状态管理与数据流

### 4.1 模块隔离原则
**对应原文：第 18 条**

- **禁止直接修改另一模块的内部状态**。跨模块通信仅允许以下方式（按推荐优先级）：
  
  1. **公共 Store（Pinia）** — 适用于全局共享数据（用户信息、权限、字典）
  2. **统一服务层** — 适用于业务逻辑复用（如 `usePermission()` composable）
  3. **事件总线（mitt / provide-inject）** — 适用于非父子但密切关联的组件通信（需在代码注释中说明事件流向）
  4. **URL Query / Route Params** — 适用于页面间参数传递
- **反模式示例**：
  ```typescript
  // ❌ 禁止：组件 A 直接修改组件 B 的响应式数据
  otherComponent.formData.name = 'xxx'
  
  // ✅ 正确：通过 Store 或事件
  userStore.updateName('xxx')
  ```

### 4.2 Pinia 状态管理
**对应原文：第 20 条**

- **全员必修**：前端团队所有人必须掌握 Pinia 的核心概念及用法：
  - `state`：定义全局响应式状态
  - `getters`：派生状态（相当于 computed）
  - `actions`：异步操作与状态变更入口
  - `$patch`：批量更新状态
  - `$subscribe`：状态变更订阅
- **Store 设计规范**：
  - 按业务域拆分 Store（如 `useUserStore`、`useDictStore`、`usePermissionStore`），禁止单一巨型 Store。
  - 每个 Store 文件行数 ≤ 200 行。
  - Actions 中禁止直接操作 DOM 或路由跳转（应在组件层处理）。
- **学习资源**：建议团队统一阅读 Pinia 官方文档核心章节，并在第一个迭代中安排一次内部分享。

### 4.3 数据流向与业务逻辑
**对应原文：第 19 条**

- **改代码前必须理清**：
  - 该模块的数据从哪里来（API → Store → Component，还是 Props 传递）？
  - 该模块的数据到哪里去（用户操作 → API 提交 → 其他模块刷新）？
  - 是否有其他页面/模块依赖该数据？
- **数据流向图**：建议对核心业务模块（如用户管理、订单流程）绘制简单的数据流向图，存于 `docs/data-flow/` 目录下。（需要团队在项目开始前绘制）

### 4.4 数据变更后的表单/列表更新
**对应原文：第 17 条**

- 增/删/改操作完成后，必须保证：
  - [ ] 当前列表/表单数据刷新
  - [ ] 相关联的其他组件数据同步更新（通过 Store 或事件通知）
  - [ ] 分页信息重置或保持合理状态
- **推荐模式**：
  ```typescript
  // 使用 provide/inject 或 emit 向上传递"需要刷新"事件
  // 或使用 Store 维护一个 refreshKey，变更时触发 watch 重新拉取
  const refreshKey = ref(0)
  async function handleDelete(id: string) {
    await deleteApi(id)
    refreshKey.value++ // 触发依赖此 key 的组件刷新
  }
  ```

- **路径清晰，避免了 CR 时的争吵**：明确了 `@/composables/` 路径，团队不用纠结“权限是放 utils 还是 hooks”。

### 2. 表格中的“坑”与“优化建议”（帮你避雷）

| 序号    | 你的表格内容          | **我的实战补充（避坑指南）**                                 |
| :------ | :-------------------- | :----------------------------------------------------------- |
| **1**   | **分页逻辑**          | 建议 `usePagination` **只管理页码和条数**，不要混入请求逻辑。否则换一个接口就要重新写一个 composable，反而臃肿。 |
| **2**   | **表格查询逻辑**      | 这个容易和 `usePagination` 耦合。**建议合并为 `useTable`**，它统一管理：`loading` + `list` + `pagination` + `searchForm` + `reset`。一个 Composable 搞定一整个表格。 |

| **8**   | **请求 Loading 逻辑** | 建议封装为 **`useRequest`**，统一管理 `loading`、`error`、`data`，并自动处理竞态问题（Race Condition）。 |

### 3. 针对请求 Loading的补全实现

既然表格留空了，我给你一个可以直接写进 `@/composables/useRequest.ts` 的 JS 版本核心思路：

```javascript
// @/composables/useRequest.ts
import { ref } from 'vue'

export function useRequest(asyncFn, options = {}) {
  const loading = ref(false)
  const data = ref(null)
  const error = ref(null)

  const execute = async (...args) => {
    loading.value = true
    error.value = null
    try {
      const result = await asyncFn(...args)
      data.value = result
      return result
    } catch (e) {
      error.value = e
      throw e
    } finally {
      loading.value = false
    }
  }

  return { loading, data, error, execute }
}
```
**用法**：页面里直接用 `const { loading, execute } = useRequest(api.getList)`，再也不用在每个接口那里单独写 `try catch` 和 `loading.value = true/false` 了。

### 5.2 Composable 编码规范

```typescript
// usePagination.ts 示例模式
export function usePagination(fetchFn: (params: any) => Promise<any>) {
  const pageNum = ref(1)
  const pageSize = ref(10)
  const total = ref(0)
  const loading = ref(false)

  const pagination = computed(() => ({
    pageNum: pageNum.value,
    pageSize: pageSize.value,
    total: total.value,
    onChange: (page: number) => { pageNum.value = page; fetchData() },
    onPageSizeChange: (size: number) => { pageSize.value = size; pageNum.value = 1; fetchData() },
  }))

  async function fetchData(extraParams?: Record<string, any>) {
    loading.value = true
    try {
      const res = await fetchFn({ pageNum: pageNum.value, pageSize: pageSize.value, ...extraParams })
      total.value = res.total
      return res
    } finally {
      loading.value = false
    }
  }

  return { pageNum, pageSize, total, loading, pagination, fetchData }
}
```

- **命名规范**：Composable 以 `use` 开头，文件名使用 `camelCase`。
- **单一职责**：每个 Composable 仅负责一类逻辑，禁止"万能 composable"。

## 六、数据反显与渲染

### 6.1 组件数据反显规范
**对应原文：第 2 条**

- **反显逻辑**（Data Back-Display）指将后端返回的数据正确地渲染到 UI 组件中，常见问题及解决方案：
  
  | 问题 | 解决方案 |
  |------|---------|
  | 字段映射不一致 | 统一在 API 响应拦截器或 Composable 中做字段映射，禁止在模板中直接写 `data.xxx !== undefined ? data.xxx : data.yyy` |
  | 空值/undefined 导致渲染异常 | 使用可选链 `?.` 和空值合并 `??`，或在 Composable 中统一兜底 |
  | 日期/金额格式不一致 | 统一封装 `formatDate()`、`formatMoney()` 工具函数 |
  | 枚举/字典值未翻译 | 使用 `useDict()` 统一转换，禁止模板中硬编码 `{1: '启用', 0: '禁用'}` |
  | 图片/文件下载 URL 未拼接完整域名 | 统一在 Axios 拦截器中处理 baseURL 拼接 |
- **自检清单**：每个有数据反显的组件开发完成后，至少构造 3 种数据场景进行验证：正常数据、空数据、异常数据。

### 6.2 字典字段集中维护
**对应原文：第 26 条**

- **集中维护位置**：Pinia Store（`useDictStore`）或常量文件（`constants/dict.ts`），二选一并在项目 README 中说明。
- **规范**：
  ```typescript
  // constants/dict.ts
  export const USER_STATUS = {
    1: { label: '启用', color: 'green' },
    0: { label: '禁用', color: 'red' },
  } as const
  
  export const ORDER_STATUS = {
    PENDING: { label: '待处理', color: 'orange' },
    PROCESSING: { label: '处理中', color: 'blue' },
    COMPLETED: { label: '已完成', color: 'green' },
  } as const
  ```
- **禁止模式**：禁止在组件模板或 `<script>` 中直接定义字典映射对象；禁止在多个文件中重复定义同一字典。

## 八、模块间连接与架构

**对应原文：第 16 条**

### 8.1 连接清晰性原则

- **页面之间**：通过路由参数和 Store 传递数据，禁止使用 `localStorage`/`sessionStorage` 传递结构化数据（仅允许存储 token、用户偏好等）。
- **模块之间**：仅通过公共 Store / 统一服务层 / 事件总线通信，禁止模块间直接 `import` 对方的组件内部方法或响应式变量。
- **组件之间**：
  - 父子：Props down, Events up
  - 兄弟：通过共同父组件或 Store 中转
  - 跨层级：`provide/inject`（需在 Provide 处注释说明下游消费者）
- **架构文档**：建议在 `docs/architecture.md` 中绘制模块依赖图，标注各模块的 Store 依赖和路由关系。

### 1. 父子通信：Props Down, Events Up（单向数据流）

- **现实比喻**：**父亲给儿子零花钱（Props Down），儿子想买鞋不能自己从父亲钱包里拿，得喊一声“爸，给点钱”（Events Up）。**
- **技术实现**：
  - **Props Down**：父组件通过 `:data="xxx"` 传值，子组件通过 `defineProps(['data'])` 接收。**数据只能从上往下流。**
  - **Events Up**：子组件通过 `emit('update', value)` 派发事件，父组件通过 `@update="handleUpdate"` 监听并修改数据。
- **⚠️ 铁律（CR 必看）**：**子组件严禁直接修改 Props 里的值！**（比如 `props.data = '新值'` 是绝对禁止的）。必须通过 `emit` 通知父组件去改，因为“谁持有数据，谁负责修改”。

---

### 2. 兄弟通信：通过共同父组件或 Store 中转

- **现实比喻**：**哥哥和弟弟想商量事，要么拉着老爸当传话筒（共同父组件），要么直接在家庭微信群聊里说（Store）。**
- **技术实现**：
  - **方案A（共同父组件中转）**：弟传消息给爸（emit），爸再传给哥（props）。**缺点**：如果嵌套层级很深，代码会极其啰嗦（Props 地狱）。
  - **方案B（Store/状态管理）**：**这是现在的主流推荐**。兄弟俩都引入 Pinia / Vuex，A 兄弟调用 `store.setMessage('xxx')`，B 兄弟直接读取 `store.message`。**解耦，且数据源唯一。**
- **⚠️ 选型建议**：如果只是传个简单的开关状态（如弹窗显隐），用方案A；如果是用户信息、权限、复杂表单数据，**无脑选方案B（Store）**。

---

### 3. 跨层级通信：provide / inject（需在 Provide 处注释说明下游消费者）

- **现实比喻**：**爷爷想给重孙子塞糖，不需要经过爸爸和儿子一层层转交，直接开个“家族内部直达通道”递过去。**
- **技术实现**：祖先组件用 `provide('key', value)` 提供数据，无论隔了多少层，后代组件直接用 `inject('key')` 获取。
- **⚠️ 最关键的“注释要求”（你贴的原文特别强调了）**：

为什么**必须在 Provide 处写注释**？
因为 **`provide/inject` 是“隐式契约”**。父组件提供了数据，但子组件（甚至孙子、曾孙）都能偷偷 `inject` 拿走。如果不在 Provide 的地方写清楚，**三个月后重构代码时，你根本不敢删这个 provide，因为你不知道下游到底有多少个组件在偷偷用它！**

**正确的代码写法（注释范例）**：

```javascript
// 在祖先组件中
import { provide, ref } from 'vue'

// ── 跨层级通信 Provide ──
// ⚠️ 下游消费者（Injector）：
// 1. @/components/DeepChild/Header.vue (用于显示用户名)
// 2. @/components/DeepChild/Profile.vue (用于修改头像)
// 若需移除或重命名，请务必同步更新上述两个文件！
const userInfo = ref({ name: '张三' })
provide('userInfo', userInfo)
```

- **🚨 红线警告**：如果删除了 `provide` 却没删下游的 `inject`，页面会直接报错崩溃，且报错信息很难定位（因为它只在运行时炸）。所以，**只提供不注释 = 埋雷**。

---

### 4. 总结

| 场景                                 | 最佳方案                          | 禁忌                                     |
| :----------------------------------- | :-------------------------------- | :--------------------------------------- |
| **父子**                             | **Props + Emit**                  | 禁止子组件直接改 Props                   |
| **兄弟**                             | **Pinia/Vuex (Store)**            | 尽量避免用父组件层层传递（除非极简场景） |
| **跨层级（祖孙）**                   | **Provide / Inject**              | **必须加注释说明下游位置**，否则别用     |
| **完全无关的全局**（如主题色、语言） | **Provide / Inject** 或 **Store** | 不用非得用 EventBus（已过时）            |

**工程化的黄金标准**：在一个 Vue 组件文件中，如果你发现 `props` 传了超过 3 层（爷->父->子->孙），就**必须**改用 `provide/inject` 或 Store

## 九、版本管理与版本标记

**对应原文：第 6 条**

### 9.1 阶段性版本发布

- **版本号规范**：采用语义化版本 `MAJOR.MINOR.PATCH`（主版本.次版本.修订版本）：
  - `MAJOR`：重大功能上线或不兼容的 API 变更
  - `MINOR`：新增向后兼容的功能模块
  - `PATCH`：Bug 修复和小优化
- **版本标记**：每次部署到服务器时，在 Git 仓库打 `tag`（如 `v1.2.0`），Release Notes 中列出该版本的：
  - 新增功能
  - 修复问题
  - 已知问题
  - 关联的需求/任务编号
- **版本目标清单**：每个版本开始开发前，必须在任务管理工具（如 TAPD / Jira / 飞书多维表格）中列出该版本的：
  
  - [ ] 目标功能清单
  - [ ] 预期完成时间
  - [ ] 负责人
  - [ ] 验收标准
  
  版本控制要点：1.具有两个主分支，服务端上的上线主分支，开发端可以随时上传修改的开发用主分支。2.在团队认为本次版本任务已经完成之后，需要在开发端主分支上对当前版本所设立的目标进行验收。验收完毕后才能上传到服务端上，视为一次版本更新。

## 十、后端部署通知

### 10.1 部署通知流程

- **后端部署时必须通知前端**，通知渠道：团队群（即时）+ 项目协作平台（留痕）。
- **通知内容至少包含**：
  - 部署时间

  - 变更内容（新增接口 / 修改接口 / 字段变更 / 数据库变更）
  - 影响范围（哪些页面/功能受影响）
  - 接口文档是否已同步更新
- **前端响应**：收到通知后 4 小时内完成对应页面的接口联调验证，如有问题即时反馈。

## 十一、风格统一与决策流程

**对应原文：第 8 条**

### 11.1 风格冲突处理机制

- **发现问题时**：任何人在项目中发现自己书写的代码风格与他人不一致时，**暂停当前的实现**，主动找前端负责人确认应采用何种风格。
- **决策记录**：确认后的风格选择写入项目 `docs/style-guide.md` 或更新 ESLint 规则，避免同类问题重复出现。
- **建议预定义风格参照**：推荐项目初始化时选定一个主流风格指南（如 Vue 官方风格指南或 AirBnB JavaScript Style Guide）作为基准，减少临时决策。

## 十二、AI 辅助开发规范

**对应原文：第 23 条**

### 12.1 使用 AI 时必须提供的上下文

前端工程师在使用 AI 辅助开发时，Prompt 中必须明确提供：

| 类别 | 必填内容 | 示例 |
|------|---------|------|
| **项目信息** | 技术栈（Vue 3 + Pinia + Element Plus + TypeScript） | ✅ |
| **文件职责** | 当前修改文件的用途、在项目中的角色 | "该文件为用户列表页，负责展示、搜索、删除用户" |
| **已有上下文** | 相关 Store / API / Composable 的名称和用法 | "用户数据来自 `useUserStore`" |
| **修改目标** | 具体要改什么、为什么改 | "在表格上方增加批量删除按钮" |
| **禁止改动范围** | 明确哪些文件/逻辑不要动 | "不要修改 `useUserStore.ts` 中的 action 逻辑" |
| **约束条件** | 命名规范、字段来源、输出格式 | "字段名必须与 `api/user.ts` 中的接口返回一致" |

### 12.2 AI 使用红线（绝对禁止）

以下行为必须杜绝，违者 Code Review 直接打回：

| 禁止行为 | 说明 |
|---------|------|
| ❌ 大范围重构无关代码 | AI 生成代码时不应改动与需求无关的文件或逻辑 |
| ❌ 随意改变已有业务逻辑 | AI 不理解业务上下文，可能"优化"掉关键判断 |
| ❌ 删除注释中的重要业务说明 | 业务注释是团队知识的载体，AI 不能判断其重要性 |
| ❌ 引入未经确认的新依赖 | 每次引入新 npm 包需团队评审 |
| ❌ 伪造接口字段 | 字段必须来自接口文档或后端实际返回 |
| ❌ 把 Mock 数据当真实字段 | Mock 仅用于开发阶段，提交时需清理或显式标注 |
| ❌ 绕过类型检查 | 禁止使用 `as any`、`@ts-ignore` 绕过 TypeScript 检查（确需例外需注释说明 + CR 审批） |

### 12.3 AI 生成代码后人工检查清单

每次 AI 生成代码后，开发者必须逐项确认：

- [ ] **是否改动无关文件？** — 使用 `git diff --stat` 检查变更范围
- [ ] **是否引入新依赖？** — 检查 `package.json` 是否新增依赖
- [ ] **字段是否对得上接口文档？** — 逐一对比 API 返回的字段名、类型
- [ ] **是否破坏原有样式？** — 运行项目，逐页对比改动前后的效果
- [ ] **是否存在重复逻辑？** — 搜索项目中是否已有相同功能的 Composable / 工具函数
- [ ] **是否能通过构建？** — 本地执行 `npm run build` 或 `pnpm build` 确认无编译错误
- [ ] **ESLint 是否通过？** — 执行 `npm run lint`

  - 项目 README 中的架构图

### 13.2 任务跟踪与阶段性总结
**对应原文：第 28 条**

- **任务文档要求**：每项任务必须记录以下字段：
  | 字段 | 说明 |
  |------|------|
  | 任务名称 | 简洁明确的描述 |
  | 负责人 | 唯一负责人 |
  | 开始时间 | 实际开始日期 |
  | 完成时间 | 实际完成日期 |

## 十四、沟通协作规范（补充）

> 以下为基于原始建议精神补充的专业化规范。

### 14.1 技术决策记录（ADR）

- 重要的技术选型、架构变更、规范调整必须通过 **ADR（Architecture Decision Record）** 记录。

- 格式：标题（决策内容）、背景、决策、后果、备选方案。

- 存储位置：`docs/adr/` 目录，按序号命名（如 `001-use-pinia-over-vuex.md`）。

- ADR 是一份简短的 Markdown 文本文件，用于记录团队在某个时间点做出的重要技术决策，并详细说明“为什么要这么做”以及“带来了什么后果”，可以视作项目技术演进的“历史档案馆”或“决策日记本”。
  为什么要写 ADR？（核心价值）
  防止“失忆”：三个月后，新来的同事问“为什么当初不用 Cypress 而用 Playwright”？你不需要去翻聊天记录或回忆当时的争吵，直接看 ADR-001 就明白了。
  明确权责：在 ADR 中可以清晰写明 “决策依据”（如“基于测试环境要求”或“依赖第三方 SDK 限制”），避免后人仅仅因为“某某大厂在用”就盲目跟风。
  作为 CR 的补充：代码审查（CR）只能看到最终的代码结果（代码是什么样的），而 ADR 解释了代码背后的架构逻辑（代码为什么要这样设计）。二者结合，项目的可维护性会大大提升。
  低成本可追溯：它存储在代码仓库（如 Git）的 docs/adr/ 目录下。当 ADR 被推翻时（比如换回 Cypress），我们不需要删除旧文档，只需新增一个 ADR-002 将旧 ADR 标记为 “已废弃（Superseded）”，并说明原因，完整保留了演进脉络。

- 参考模板：
  # ADR 001：选用 Playwright 作为 E2E 测试框架
  ## 背景（Context）
  项目需要自动化测试核心登录和下单流程。现有 Cypress 版本较旧，且 Safari 兼容性测试需求日益增多。
  ## 决策（Decision）
  团队决定在新项目中采用 Playwright 替代 Cypress 作为 E2E 测试框架。
  ## 理由（Rationale / 为什么）
  - Playwright 原生支持 Safari（WebKit），满足多端需求。
  - 执行速度比现有 Cypress 配置快 30% 以上。
  - 支持多标签页场景，更贴合业务的 OAuth 登录流程。
  ## 后果（Consequences）
  - **正面**：CI 门禁中的测试阶段时间缩短，测试稳定性提高。
  - **负面**：团队成员需要花费 2 天时间学习 Playwright 的新 API 语法。
  - **风险**：老项目仍保留 Cypress，团队短期内需维护两套测试工具。

### 14.2 Code Review 检查清单

Code Review 时除功能正确性外，必须检查以下维度：

- [ ] 文件命名是否符合规范（参考 1.1）
- [ ] 单文件行数是否超限（参考 2.1）
- [ ] 是否有未经沟通的重复实现（参考 2.2）
- [ ] 是否有硬编码的内联样式（参考 1.3）
- [ ] 接口字段是否与文档一致（参考 3.4）
- [ ] 弹窗是否有完整的 Loading 态（参考 3.3）
- [ ] 是否跨模块直接修改状态（参考 4.1）
- [ ] 字典是否集中维护（参考 6.2）
- [ ] 是否使用了 `as any` / `@ts-ignore`（参考 12.2）
- [ ] AI 生成的代码是否经过人工检查（参考 12.3）

### 14.3 知识共享机制

- 每个迭代至少安排一次 **团队内技术分享**（30 分钟以内），内容可以是：
  - 新掌握的技能/工具
  - 踩过的坑与解决方案
  - 对现有规范的改进建议
- 新人入职时，须在 3 天内通读本文档并在任务管理工具中确认签字。
